# ESPHome Smart Updater — Blueprints

This repository provides ready-to-use Home Assistant blueprints for the  
**ESPHome Smart Updater** integration.

⚠️ These blueprints require the integration:  
👉 https://github.com/PaulBiod/ha-esphome-smart-updater

---

# 📦 Available blueprints

## 1. Scheduled Safe Campaign

Automatically start update campaigns with safety checks.

👉 **Add directly to Home Assistant:**  
[![Open your Home Assistant instance and import the blueprint](https://my.home-assistant.io/badges/blueprint_import.svg)](https://my.home-assistant.io/redirect/blueprint_import/?blueprint_url=https://raw.githubusercontent.com/PaulBiod/ha-esphome-smart-updater/main/blueprints/automation/esphome_smart_updater/scheduled_safe_campaign.yaml)

👉 **Raw import link:**  
https://raw.githubusercontent.com/PaulBiod/ha-esphome-smart-updater/main/blueprints/automation/esphome_smart_updater/scheduled_safe_campaign.yaml

### Features

- Scheduled execution
- Minimum updates threshold
- Skip `.0` releases
- Version gap protection
- Optional start notification (fully customizable)

# 🧠 Available variables (Blueprint #2)

These variables can be used inside any action field:

| Variable | Description |
|----------|------------|
| `esu_result` | Campaign result (informational only) |
| `esu_done` | Number of successful updates |
| `esu_failed` | Number of failed updates |
| `esu_skipped` | Number of skipped updates |
| `esu_unavailable` | Devices unavailable / unknown status |
| `esu_last_report` | Full report generated by ESU |

⚠️ **Important:**  
`esu_result` is informational only.  
For logic/conditions, always use `esu_failed`.

# 🔧 Examples (Blueprint #2)

## 📱 Mobile / Telegram notification

```yaml
action: notify.mobile_app_your_phone
data:
  title: "ESPHome Smart Updater: {{ esu_result }}"
  message: |
    ✅ Success: {{ esu_done }}
    ❌ Failed: {{ esu_failed }}
    ⏭ Skipped: {{ esu_skipped }}
    ⚫ Unavailable: {{ esu_unavailable }}

    {{ esu_last_report }}
```

---

## 🔊 Alexa announcement

```yaml
action: notify.alexa_media_your_echo
data:
  message: >
    ESPHome update campaign finished.
    {{ esu_done }} succeeded,
    {{ esu_failed }} failed,
    {{ esu_skipped }} skipped,
    and {{ esu_unavailable }} unavailable.
  data:
    type: announce
```

---

## 🔊 Google TTS

```yaml
action: tts.speak

  entity_id: tts.google_translate_en_com
data:

  message: >
    ESPHome update finished.
```

# 🧠 Design philosophy

- Blueprints decide **WHEN** and **WHAT TO DO**
- The integration decides **HOW updates are executed**

This ensures:
- no duplicated logic
- full compatibility with future integration updates
- maximum flexibility for users

---

# 🚀 Tip

You can combine both blueprints:

- Blueprint #1 → automatically start campaigns  
- Blueprint #2 → react to results (notify, alert, automation, etc.)
